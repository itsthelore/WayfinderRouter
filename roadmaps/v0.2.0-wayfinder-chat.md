# Roadmap: v0.2.0 — `wayfinder-chat` (turnkey chat + routing)

## Status

Planned

The direction is decided in WF-ADR-0010. This roadmap sequences the first
milestone. It spans **two repositories** and therefore two version lines:

- **`wayfinder-router` v0.2.0** — a small, additive gateway enabler.
- **`wayfinder-chat` v0.1.0** — a new repository: a fork of
  [LibreChat](https://github.com/danny-avila/LibreChat) (MIT) that depends on the
  Wayfinder gateway.

Scope choice (decided): ship the **full experience** in this milestone — not just
a preconfigured bundle, but the routing-threshold control in LibreChat's settings
and the "which model answered" display in the chat.

## Context

Today Wayfinder is transparent middleware you put behind your own OpenAI-compatible
client (WF-ADR-0004). That serves power users; it does not serve casual users who
want one thing to install and chat with, routing built in. WF-ADR-0010 decides the
turnkey experience ships as `wayfinder-chat`, a LibreChat fork that *depends on*
`wayfinder-router` — keeping the router lean and UI-less while a separate product
delivers chat + routing.

LibreChat's extension points cover the *integration* (a custom endpoint pointed at
the gateway, and coarse routing via `auto` / `prefer-local` / `prefer-cloud` model
names) but **not** the *routing UX*: a real threshold control in settings and
surfacing the routing decision both need LibreChat source changes — which is why
`wayfinder-chat` is a source fork, not a thin bundle (WF-ADR-0010).

## Outcomes

- A one-command turnkey stack (`docker compose up`) brings up LibreChat + the
  Wayfinder gateway + a local model, giving a chat window with routing built in.
- The local-vs-cloud **threshold is adjustable from the chat settings** and takes
  effect per conversation.
- Each answer shows **which model handled it** (local/cloud) and its score.
- `wayfinder-router` and `wayfinder-chat` install and run independently; the
  router's identity (WF-ADR-0001/0004/0005) is unchanged beyond an additive
  override.

## Initiatives

Sequenced; later initiatives depend on earlier ones.

### Initiative 1 — Gateway per-request routing override (`wayfinder-router` v0.2.0)

The enabler, and the gating dependency for the UI work. Today the gateway routes
off the `wayfinder-router.toml` threshold; it must also accept a **per-request
override** so the chat UI can change routing per conversation:

- Accept an override on the request (transport TBD — see Open Questions): a
  threshold and/or a model-pair, and/or a model-name mapping
  (`auto` / `prefer-local` / `prefer-cloud` → thresholds).
- Hooks into the existing scoring + forward path in `wayfinder_router/gateway.py`;
  it changes only *which threshold/decision applies*, never adds model invocation
  or provider logic to the core (WF-ADR-0001/0004 boundary holds).
- Confirm/extend the decision signal the gateway already emits
  (`x-wayfinder-router-model` / `x-wayfinder-router-score`) so a client can display
  it; decide whether the decision also needs to be in the response body.
- Boundary + override tests; docs; a `wayfinder-router` minor release.

### Initiative 2 — Stand up `itsthelore/wayfinder-chat` (fork LibreChat)

- Create the repo as a **fork of LibreChat** (MIT); set an `upstream` remote and a
  branch strategy that keeps the Wayfinder customization on top of trackable
  upstream (so future LibreChat updates merge cleanly).
- Preconfigure the integration with **no source changes**: a `librechat.yaml`
  custom endpoint pointed at the gateway, plus the `auto` / `prefer-local` /
  `prefer-cloud` models for coarse routing.
- Ship the **turnkey stack**: a `docker compose` that runs LibreChat + the gateway
  (`wayfinder-router`) + a local model (e.g. Ollama), wired together.
- README, branding pass (later/optional), and licence reconciliation: the fork
  stays MIT (LibreChat) with `NOTICE`/attribution; `wayfinder-router` remains
  Apache-2.0 as a dependency (both permissive, compatible).

### Initiative 3 — Spike LibreChat's client (de-risk before the source changes)

Before Initiatives 4–5, confirm against the real LibreChat code where the two
source changes hook in: the settings/parameters components (to add a control) and
the request path (to send the override), and whether the client can read response
headers (to display the decision). WF-ADR-0010 flags this as required.

### Initiative 4 — Routing-threshold control in settings (LibreChat source)

- Add a threshold control to the settings/parameters UI (LibreChat's `customParams`
  cannot define a new control), wired to send the override to the gateway per
  Initiative 1's contract.
- Keep the change **thin and localized** to a small set of client components, to
  ease upstream tracking.

### Initiative 5 — Surface the routing decision in chat (LibreChat source)

- Show which model answered (local/cloud) and the score per message, from the
  `x-wayfinder-router-*` signal. Another localized client change.
- Package + document the turnkey stack; cut `wayfinder-chat` v0.1.0.

## Constraints

- **Independence (WF-ADR-0010):** two products, used independently. `wayfinder-router`
  stays UI-less; its only change here is the additive override.
- **Deterministic boundary (WF-ADR-0001/0004):** the override changes the
  threshold/decision only — it never adds model invocation, provider selection, or
  credential handling to the core.
- **Thin fork:** keep LibreChat source changes minimal and localized so upstream
  merges stay tractable.

## Non-Goals

- Changing `wayfinder-router`'s deterministic scorer.
- Building a chat UI from scratch (the reason to fork LibreChat).
- Auth, multi-tenant, or hosting concerns beyond what LibreChat already provides.

## Open Questions

- **Override transport:** request body field vs. HTTP header vs. model-name
  mapping — which does LibreChat make easiest to send per conversation, and which
  keeps the gateway OpenAI-compatible? (Resolved by Initiative 3.)
- **Decision display source:** can LibreChat's client read the
  `x-wayfinder-router-*` response headers, or must the gateway also put the
  decision in the response body?
- **`wayfinder-chat` versioning:** start at `0.1.0` independent of LibreChat's
  version, with upstream tracked via the fork.

## Success Measures

- `docker compose up` yields a working chat window with routing; the threshold is
  adjustable in settings and changes behaviour; the answering model is visible per
  message.
- `wayfinder-router` and `wayfinder-chat` are each installable/runnable on their
  own.
- The fork's source delta is small and localized; an upstream LibreChat update can
  be merged without major rework.

## Risks

- **Fork drift / upstream churn** around the touched components. Mitigation: spike
  first (Initiative 3), keep changes thin/localized, pin and track upstream.
- **Boundary creep:** the per-request override must not pull inference into the
  core. Mitigation: it only sets the threshold/decision; reviewed against
  WF-ADR-0001/0004.
- **Sequencing:** Initiatives 4–5 depend on Initiative 1 (the gateway must honour
  the override) and Initiative 3 (where to hook in).
